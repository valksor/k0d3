# Godot Rust GDExtension — Packaging & Distribution

Linked from `Skill(godot)`. The summary lives in the main skill. Use this reference for the full cross-compile + CI + hot-reload + gotchas workflow when packaging a Rust GDExtension.

**Iron Law: `compatibility_minimum` in the `.gdextension` must match the Godot minor the `gdext` crate was built against. Mismatched manifests fail silently — the editor logs nothing useful, the class just doesn't register. Every release pins both.**

**Versions:** Godot `4.6` · `gdext 0.4.x` — _No LTS series for either; pin both in tandem. Bumping Godot minor (4.5→4.6) requires rebuilding the extension against the matching `api-4-6` feature._

## `.gdextension` manifest

Lives in the Godot project (not the Rust crate) — typically `godot-project/mygame.gdextension`:

```ini
[configuration]
entry_symbol            = "gdext_rust_init"
compatibility_minimum   = 4.5
reloadable              = true

[libraries]
linux.debug.x86_64      = "res://../rust/target/debug/libmygame_rust.so"
linux.release.x86_64    = "res://../rust/target/release/libmygame_rust.so"
macos.debug             = "res://../rust/target/debug/libmygame_rust.dylib"
macos.release           = "res://../rust/target/release/libmygame_rust.dylib"
windows.debug.x86_64    = "res://../rust/target/debug/mygame_rust.dll"
windows.release.x86_64  = "res://../rust/target/release/mygame_rust.dll"

[dependencies]
# Bundled dylibs the extension itself loads (rare — most Rust deps statically link)
```

| Field                   | Meaning                                                                                                                                                               |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `entry_symbol`          | Generated by `#[gdextension]`. Default is `gdext_rust_init` for the `godot` crate. **Do not rename** unless you also override `ExtensionLibrary::entry_symbol`.       |
| `compatibility_minimum` | Godot refuses to load if the running editor is older than this. Set to the lowest Godot minor your `gdext` feature flag supports (`api-4-5` → `4.5`).                 |
| `reloadable`            | Lets the editor unload/reload the library when Cargo rebuilds. Required for hot-reload; set `false` for production builds to avoid teardown bugs.                     |
| `[libraries]` keys      | `<platform>.<config>.<arch>` — config and arch are optional but specific keys beat general. `macos.release` covers both architectures if you ship a universal binary. |

Paths use `res://` and are relative to the Godot project root. Bare filenames (no path) cause the editor to search OS library paths — almost never what you want.

## Cross-compile matrix

| OS              | Triple                     | Notes                                                                                                                                   |
| --------------- | -------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| Linux x86_64    | `x86_64-unknown-linux-gnu` | Build on Linux. Cross from macOS via `cross`.                                                                                           |
| macOS arm64     | `aarch64-apple-darwin`     | Build on Apple Silicon.                                                                                                                 |
| macOS x86_64    | `x86_64-apple-darwin`      | Build on Apple Silicon with the toolchain installed: `rustup target add x86_64-apple-darwin`.                                           |
| macOS universal | (merge of the two)         | `lipo -create -output libmygame_rust.dylib <arm64.dylib> <x86_64.dylib>`. Then a single `macos.release =` entry.                        |
| Windows x86_64  | `x86_64-pc-windows-gnu`    | Build on Linux/macOS via `mingw-w64`. `x86_64-pc-windows-msvc` requires the MSVC toolchain — Linux-cross-to-MSVC is painful, use `gnu`. |

Install targets once:

```sh
rustup target add aarch64-apple-darwin x86_64-apple-darwin x86_64-pc-windows-gnu x86_64-unknown-linux-gnu
```

## Build coordination — `cargo-make`

A `Makefile.toml` keeps the multi-target build + copy + lipo logic out of CI YAML:

```toml
[tasks.build-all]
dependencies = ["build-linux", "build-macos-universal", "build-windows"]

[tasks.build-macos-universal]
script = '''
cargo build --release --target aarch64-apple-darwin
cargo build --release --target x86_64-apple-darwin
mkdir -p target/release-universal
lipo -create -output target/release-universal/libmygame_rust.dylib \
  target/aarch64-apple-darwin/release/libmygame_rust.dylib \
  target/x86_64-apple-darwin/release/libmygame_rust.dylib
'''

[tasks.build-windows]
command = "cargo"
args    = ["build", "--release", "--target", "x86_64-pc-windows-gnu"]
```

Plain `Makefile` works too. The point: one entry point, reproducible locally and in CI.

## Hot-reload during dev

1. Set `reloadable = true` in the manifest.
2. `cargo build` (debug). The shared library at `target/debug/...` is overwritten.
3. Godot detects the mtime change and reloads the library.
4. Open scenes that instantiate the class continue to work; existing instances rebuild their Rust state via `init()`.

Caveats:

- State that doesn't survive a reload: anything held in static `OnceLock` / `lazy_static`. Treat reload as a process restart for module-level state.
- If the API surface (`#[func]`, `#[signal]`) changes shape, save your scene first — Godot may de-link the script binding.
- Editor crashes on reload usually mean a `Gd<Node>` was stored in a static; the post-reload pointer dangles. Reproduce by quitting + reopening Godot.

## CI matrix — GitHub Actions sketch

```yaml
jobs:
  build:
    strategy:
      fail-fast: false
      matrix:
        include:
          - { os: ubuntu-latest, target: x86_64-unknown-linux-gnu, artifact: libmygame_rust.so }
          - { os: macos-latest, target: aarch64-apple-darwin, artifact: libmygame_rust.dylib }
          - { os: macos-latest, target: x86_64-apple-darwin, artifact: libmygame_rust.dylib }
          - { os: windows-latest, target: x86_64-pc-windows-msvc, artifact: mygame_rust.dll }
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          targets: ${{ matrix.target }}
      - run: cargo build --release --target ${{ matrix.target }}
      - uses: actions/upload-artifact@v4
        with:
          name: ${{ matrix.target }}
          path: target/${{ matrix.target }}/release/${{ matrix.artifact }}
```

A downstream `release` job downloads all artifacts, runs `lipo` on the two macOS binaries, and assembles a release zip with the `.gdextension` manifest pre-filled. See `Skill(ci-github-actions)` for matrix patterns + caching.

## `.gdignore` — keep Rust sources out of Godot's import scan

In the Rust crate root (or anywhere Godot's importer would otherwise wander):

```
# rust/.gdignore
```

An empty file is enough. It tells Godot "do not import anything under this directory" — saves seconds on every editor open and avoids spurious `.import` files appearing in `target/`. Always add one to `target/` and to the Rust crate root.

## Distribution paths

| Channel                   | Use when                                                                                                                                                                    |
| ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Bundled with the game** | The default. Ship the `.gdextension` + compiled libraries inside the exported project. Players never know it's a GDExtension.                                               |
| **Godot Asset Library**   | The extension is a reusable tool/plugin (editor utilities, custom nodes others embed). Asset Library wants source + prebuilt libraries; tag a release on GitHub and submit. |
| **Custom installer**      | The extension links to a system dylib (e.g., a proprietary physics engine). Bundle is a license issue; ship a downloader instead.                                           |

Export presets in Godot pick up the `.gdextension` automatically when the project is exported; double-check the export console for `Found GDExtension: ...` lines.

## Common gotchas

| Symptom                                             | Cause                                                                                                                                                                                                                |
| --------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Class doesn't appear in "Add Node" dialog; no error | `compatibility_minimum` higher than the running editor, OR `entry_symbol` typo, OR `.gdextension` not picked up by the editor's filesystem dock. Force re-scan: Project → Reload Current Project.                    |
| Class loads in editor but not in exported game      | Missing `[libraries]` entry for the export platform. Editor uses `linux.debug.x86_64`; the export uses `linux.release.x86_64`.                                                                                       |
| "Method `init` not found" at runtime                | `#[derive(GodotClass)]` without `#[class(init)]` AND no manual `init` in the `INodeXXX` impl. Pick one.                                                                                                              |
| Hot-reload crashes the editor                       | Static state holding `Gd<Node>` across reload, OR a panic in `init()` on the reloaded instance. Disable `reloadable` to confirm.                                                                                     |
| Library 2× larger than expected                     | Debug build symbols. `strip` the release library or set `[profile.release] strip = "symbols"` in `Cargo.toml`.                                                                                                       |
| Crashes on a different CPU than dev machine         | `RUSTFLAGS=-C target-cpu=native` in your shell env. **Never set `target-cpu=native` for release builds** — the binary uses instructions only your CPU has. Use `target-cpu=x86-64-v2` if you want a modern baseline. |
| Godot logs `Can't open dynamic library`             | Path in `[libraries]` doesn't resolve; the `lib` prefix is required on Linux/macOS, forbidden on Windows. Check filename exactly.                                                                                    |

## Anti-patterns

- One giant `.gdextension` for multiple unrelated tools — split per concern; each cdylib is independent.
- Committing `target/` to the Godot project — `.gitignore` it, build artifacts belong in CI artifacts or a separate sibling directory.
- Hand-editing the `.gdextension` per platform during release — script it, or `cargo-make` it. Drift causes the "loads on my machine" bug.
- `target-cpu=native` in `[profile.release]` (see gotcha above).
- Building debug + shipping it — debug binaries are 5-10× larger and slow, and gdext debug-assertion panics surface in player logs.
- Forgetting `.gdignore` in the Rust crate root — Godot creates `.import` files inside `target/` and the editor takes seconds longer to open.
- Mixing `gdnative` (v3) and `gdext` (v4) symbols in one extension — the ABIs are unrelated.
