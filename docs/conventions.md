# k0d3 conventions

Authoritative reference for skill, agent, command, and pack frontmatter. The validators in `scripts/_validate_skills.py` and `hooks/validate-skill-frontmatter.sh` enforce these rules; any divergence between this doc and the validators should be treated as a bug.

## Skill frontmatter

Every `skills/<slug>/SKILL.md` MUST open with this YAML frontmatter:

```yaml
---
name: <kebab-case-slug>
description: One-line trigger description used by Claude to decide when to invoke.
metadata:
  type: core | language | runtime | database | domain | protocol | ci | observability
    | tooling | meta | pack
  languages:
    - python
  status: draft | active
  blocked_on: Postgres 18 GA
  related:
    - <slug>
  owns: <topic>
  invokes_shell: false
  shell_reviewed: valksor 2026-05-17
---
```

### Rules

- **Slug == directory name**, always. Validator refuses commits that break this.
- **Slugs are stable once `status: active`.** `draft` skills can be renamed (sweep updates `related:` references).
- **No `status: stub`.** Validator fails. Full content is the rule; `draft` is the only "in-progress" state.
- **Body cap ~200 lines.** Long-form material goes in `references/<topic>.md` and is linked. Exception: `skills/skill-discovery/SKILL.md` is auto-generated by `scripts/generate-skill-graph.sh` (full topic→slug routing table) and is body-cap-exempt in both `scripts/_validate_skills.py` and `hooks/validate-skill-frontmatter.sh`.
- **Description cap — one-line trigger.** Only `description` (and `name`) count toward Claude Code's skill-listing budget (`skillListingBudgetFraction`, ~1% of context); the body does not. Keep `description` to a single trigger sentence carrying the routing keywords, and push detail into the body or `references/`. `scripts/_validate_skills.py` **warns at >180 chars and fails at >220**; `hooks/validate-skill-frontmatter.sh` mirrors the **>220 hard fail** at write-time (the 180 warn is lint-only). It also warns if the catalogue's total name+description footprint exceeds ~22,000 chars (the listing then risks overflowing for end users — the harness drops descriptions and nags). To keep a routing keyword that trimming removed from a description, add it to `metadata.keywords` and rerun `scripts/generate-skill-graph.sh`.
- **Cross-refs use slugs, not paths.** `related: [python-testing]`, never `related: [skills/python-testing/]`.
- **Domain owns concept; language owns dialect.** A language skill must link upward (e.g., `python-testing.related: [testing-unit]`) instead of redefining the concept.
- **`invokes_shell` is a reviewer attestation, not a factual descriptor**: setting `invokes_shell: false` AND `status: active` AND omitting `shell_reviewed:` makes `validate-skills.sh` **fail**. The flag means "an author has read every Bash invocation in this skill and attests they are safe" — not "this skill contains no shell commands." A skill with undeclared Bash content masquerading as `invokes_shell: false` is actively misleading to any caller that uses the flag to assess trust. Setting `invokes_shell: true` produces only a warn (the risk is self-evident).
- **Skills brought in from outside** default to `invokes_shell: true` and stay `draft` until you read their Bash invocations line-by-line, then flip `invokes_shell: false` with a `shell_reviewed:` stamp.

## Agent frontmatter

```yaml
---
name: <agent-name>
description: One-line trigger description.
tools: [Read, Edit, Write, Grep, Glob, Bash]
model: sonnet | opus | haiku | inherit
expertise: language | code-quality | review-perspective | workflow | domain
skills: [<slug>, <slug>] # skills this agent invokes
---
```

### Field values

- **model**: `sonnet`, `opus`, `haiku`, or `inherit` (use the parent session's model — appropriate for read-only review agents that should run on whatever the user is already paying for).
- **expertise**: `language` (e.g., python-expert), `code-quality` (read-only review cohort), `review-perspective` (the four calibrated reviewers), `workflow` (process/meta agents), `domain` (cross-cutting domain experts like security-auditor, frontend-designer).
- **tools**: standard set is `[Read, Edit, Write, Grep, Glob, Bash]`. Additional grants:
  - `WebSearch` — for agents that need to look up external info (e.g., `error-whisperer`, `unsticker`). Surface to the user that WebSearch must be enabled in their CC session for the agent to function fully; the agent should degrade gracefully if absent.
  - `Bash(<prefix>:*)` — scoped bash. Matches commands whose first token has the given prefix. `Bash(git:*)` matches ANY git subcommand including writes (`push`, `reset --hard`, `clean -fd`). If a read-only scope is required, add an explicit "Tool scope" section to the agent body listing the allowed subcommands.
  - `Edit` / `Write` — granted unscoped. If the agent should write to a narrow set of paths, document the restriction in the body's "Tool scope" section.

### Lint behavior

The lint gate parses agent body for `Skill(<slug>)` calls **outside fenced code blocks** and warns when they diverge from the `skills:` array.

- Conditional prose invocations count (e.g., "If repo is Python, invoke `Skill(python-idioms)`").
- Example invocations inside ` ``` ` fences do NOT count (documentation, not execution).
- **Caveat**: indented-only code blocks (4-space) are NOT excluded from lint. Use fenced blocks for examples.
- **Code-quality review agents** (`code-reviewer`, `code-simplifier`, `silent-failure-hunter`, `comment-analyzer`, `type-design-analyzer`, `pr-test-analyzer`) drop `Bash`, `Edit`, and `Write` from `tools:`. Read-only review only.
- **Calibrated reviewer agents** (`reviewer-senior-dev`, `reviewer-senior-qa`, `reviewer-security`, `reviewer-end-user`) also drop `Bash`/`Edit`/`Write` for the same reason — they review, they do not modify.

## Command frontmatter

```yaml
---
name: <command-name>
description: One-line trigger description.
argument-hint: "[args]"
allowed-tools: [Read, Grep, Glob, Bash, Agent]
---
```

## CLAUDE.md content

Required content (see repo-root `CLAUDE.md`):

1. One-paragraph "what is k0d3" intro
2. **Prominent**: "Invoke `Skill(using-k0d3)` at session start."
3. Slug convention summary
4. Navigation map (architecture.md + skill-graph.md)
5. One-skill-per-use rule (no packs, no multi-chunk splits of a single topic)
6. Prefix rule: use the explicit `k0d3:` prefix to disambiguate from other installed plugins
7. Tool permissions block

## References naming

`<topic>-<version?>.md`, kebab-case, no dates in filename. Versions only when content is version-specific (e.g., `postgres.md`, `wcag-2.1-aa-checklist.md`).

## Slug naming patterns

| Family         | Pattern                         | Examples                                           |
| -------------- | ------------------------------- | -------------------------------------------------- |
| Core / process | bare topic                      | `tdd`, `debugging`, `commit-writer`                |
| Language       | `<lang>-<topic>`                | `go-idioms`, `python-async`, `ts-strict-mode`      |
| Runtime        | `<runtime>-essentials`          | `bun-essentials`, `pnpm-essentials`                |
| Database       | `<db>-<topic>` or `sql-<topic>` | `postgres`, `sql`                                  |
| Domain         | `<domain>-<topic>`              | `security`, `frontend-tailwind`                    |
| Game dev       | `game-dev-<topic>`              | `game-dev-godot-signals`                           |
| Protocol       | `<protocol>-essentials`         | `rest-essentials`, `graphql-essentials`            |
| CI             | `ci-<platform>`                 | `ci-github-actions`, `ci-gitlab-ci`                |
| Observability  | `observability-<topic>`         | `observability-essentials`, `observability-sentry` |
| Tooling        | `tooling-<tool>`                | `tooling-jq`, `tooling-ripgrep`                    |

## Validator bypass

If `hooks/validate-skill-frontmatter.sh` blocks a needed write:

| Method                           | Command                                                         | When to use                                                                                                                                                    |
| -------------------------------- | --------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **One-shot (preferred)**         | `env K0D3_SKIP_VALIDATOR=1 claude`                              | Single Claude session/invocation. Bypass is **logged** to `.claude/logs/validator-bypass.log`, leaving an audit trail.                                         |
| Per-session (fish)               | `set -x K0D3_SKIP_VALIDATOR 1` then launch Claude in that shell | One Claude session. Same audit-trail behavior as the one-shot form.                                                                                            |
| Persistent disable (last resort) | `chmod -x hooks/validate-skill-frontmatter.sh`                  | Long sweep of skill writes you've already validated. **No audit trail.** Re-enable immediately afterwards with `chmod +x hooks/validate-skill-frontmatter.sh`. |

The env var must be set **in the shell that launches Claude Code**, not in a subshell. Subshells do not propagate to hook subprocesses unless exported.

The hook is **fail-open by design**: if PyYAML is missing, the validator script has any internal error, or the bypass fires, the hook exits 0 and allows the write. A failed validation is logged and surfaced via stderr but does not block. Treat the validator as a developer-aid tripwire, not a security gate.

## Re-smoke after status flip

When a skill transitions from `draft` → `active`, run `bash scripts/smoke-skills.sh` to include it in the next smoke run. The script only iterates `status: active` skills (drafts are skipped by design — documented in the script header).

## Lint warning conditions (quick reference)

- `status: stub` → **fail**
- Slug ≠ directory name → **fail**
- Body line count > 200 (and not `skills/skill-discovery/`) → **fail**
- `description` > 220 chars → **fail** (write-time hook mirrors this hard cap)
- `description` > 180 chars → warn (tighten toward a one-line trigger)
- Total skills `name`+`description` footprint > ~22,000 chars → warn (session skill-listing may overflow `skillListingBudgetFraction`)
- `related:` slug doesn't resolve → **fail**
- Missing `metadata.type` or `metadata.status` → **fail**
- `invokes_shell: false` + `status: active` + no `shell_reviewed:` → **fail** (attestation required for the load-bearing "no shell" claim)
- `invokes_shell: true` (any status) → **warn** (risk is obvious; manual review still required before flipping to active)
- `blocked_on:` date in the past → warn
- Agent `Skill(<slug>)` in body diverges from frontmatter `skills:` → warn
- `last-generated:` on `skill-discovery` older than newest skill → warn
